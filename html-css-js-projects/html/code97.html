Code 97: <noscript>

The <noscript> element defines alternate content for users who have JavaScript disabled or if the browser doesn't support it. Content inside only renders when JS can't run.

Examples

1. Basic JS warning

<noscript>
  <p>Please enable JavaScript to use this site.</p>
</noscript>
Shows only when JS is off or blocked

2. Fallback for JS-powered image gallery

Loading gallery...

3. Form without JS validation

4. iframe for tracking pixels

<!-- Analytics script -->
<script src="analytics.js"></script>

<!-- Fallback for no-JS tracking -->
<noscript>
  <img src="/track?noscript=1" width="1" height="1" alt="">
</noscript>
Common pattern for analytics when JS fails

5. noscript in head for CSS

<head>
  <noscript>
    <link rel="stylesheet" href="no-js.css">
  </noscript>
</head>
Loads alternate CSS only when JS is disabled

7. What NOT to do

Bad - hiding content that doesn't need JS:

<noscript>
  <style>body { display: none; }</style>
  Enable JS to view this page
</noscript>
Don't block content unnecessarily. Use progressive enhancement instead

Note

noscript purpose: <noscript> provides fallback HTML when scripts can’t run. Browsers with JS enabled completely ignore its contents.

Two contexts: In <head>, <noscript> can only contain <link>, <style>, and <meta>. In <body>, it can contain any flow content.

When it triggers: 1. User disabled JS. 2. Browser doesn't support <script>. 3. Corporate firewall strips JS. It does NOT trigger if JS errors out.

Progressive enhancement: Best practice is to build core functionality in HTML/CSS first, then layer JS on top. Use <noscript> for truly JS-dependent features.

SEO: Googlebot runs JavaScript, but <noscript> content is still indexed. Don’t stuff keywords here - it looks spammy.

Usage today: ~0.2% of users have JS disabled. But <noscript> is still useful for tracking pixels, print styles, and accessibility.

Can't detect with JS: Obviously you can’t use JS to check if <noscript> ran. It’s purely a browser fallback mechanism.
